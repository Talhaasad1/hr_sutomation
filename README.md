# ATS Portal — Industry-Level HR Automation System

A full **Applicant Tracking System**: a public Career Portal where candidates
apply with just their name, email, phone, and resume — and a MongoDB-backed
HR/Recruiter/Admin portal with AI resume screening, automatic ranking, an ATS
pipeline, interview scheduling with calendar invites, automatic (editable)
emails, an offer letter generator, analytics dashboards, and an admin panel.

## How it's meant to be used
- **Candidates never log in.** They open the Career Portal, pick a job, and
  submit Name + Email + Phone + Resume (PDF/DOCX). That's it.
- **HR/Recruiters/Admins log in** to the same app (via "HR / Staff Login") to
  review, screen, rank, move candidates through the pipeline, schedule
  interviews, and send emails/offers.

## 🆕 Recent Updates
- **Candidate Pipeline**: added a Phone column, a Score Threshold slider to filter candidates, and a one-click
  **"Auto-Screen 'Applied' Candidates"** action — anyone at/above the threshold is auto-Shortlisted, anyone below
  is auto-Rejected. This **only** touches candidates whose current status is still `Applied`; anyone already
  Shortlisted/Rejected/Interviewing/etc. is left untouched.
- **Interview Scheduler**: now also has a **Status-based Email Queue** — candidates are classified by their
  current pipeline status (Shortlisted, Rejected, etc.) so you always know exactly which email needs to go to
  whom, with a pick-list and a "Send to Selected" action. Already-emailed candidates for that stage are excluded automatically.
- **Candidate Profiles**: a dedicated page — filter by Job Applied / Name / Status, then open any candidate to
  see their full profile (resume data, application + interview history) and **download their original CV**.
- **Branding & AI settings are now Admin-only**, moved into the Admin Panel (a "Branding & AI" tab) — HR/Recruiter
  accounts no longer see this page.
- **AI now performs the actual evaluation**: Match Score, Strong/Missing Skills, Weak Areas, and Recommendation
  are generated by Claude (via the Admin-configured Anthropic API key) at the moment a candidate applies. If no
  key is configured (or the AI call fails), it automatically falls back to the TF-IDF + skill-overlap evaluator
  so the system always produces a score.
- Every submitted CV is now saved to disk (`assets/resumes/`) and linked to the candidate record, and a full
  candidate profile document is saved in MongoDB the moment someone applies — both viewable and downloadable
  from Candidate Profiles.

## 🆕 Industry-Hardening Update (Security, Scale, Ops)

Built specifically for a small team (5-10 users) — meaning: real protection
where it matters, without adding infrastructure (Redis, Celery, etc.) that's
unnecessary at this scale and would just be more things that could crash or
be attacked.

### 🔒 Security
- **Brute-force lockout**: 5 failed logins locks that username out for 15 minutes.
- **Forced password change**: the seeded `admin`/`admin123` account must set a
  new password on first login — it can't be used long-term by accident.
- **Stored-XSS hardening**: every place a job title/description, company name,
  or other free-text field gets rendered as HTML is now escaped
  (`ui.esc()`), so nobody can inject a `<script>` tag via a job posting or
  branding field.
- **File upload validation**: resumes are checked for actual PDF/DOCX magic
  bytes (not just the filename extension) and capped at 8MB — this applies
  to both the single apply flow and bulk ZIP imports.
- **Defense-in-depth RBAC**: the Admin Panel explicitly checks the logged-in
  user's role in code, not just by hiding the nav button.
- **Auto-expiring sessions**: login tokens and lockout records use MongoDB
  TTL indexes, so stale data cleans itself up — no manual cron needed for that.
- **Non-root Docker container**: the app runs as an unprivileged `appuser`
  inside its container.

### ⚡ Background Processing (lightweight — no Redis/Celery)
- **`worker.py`**: a standalone process that polls a `background_tasks`
  collection in MongoDB and processes jobs like bulk resume imports — so
  importing hundreds of resumes never freezes the HR portal's UI.
  MongoDB itself is the queue; at 5-10 users' scale this is simpler and has
  a smaller attack surface than running a separate Redis + Celery stack.
  Run it with `python worker.py` alongside the app (or as its own Docker
  service — already wired up in `docker-compose.yml`).

### 📦 Bulk Resume Import
- **Job Management → 📦 Bulk Resume Import**: upload a `.zip` of existing
  PDF/DOCX resumes, pick the target job, and they're all parsed, screened,
  and ranked automatically in the background, with a live progress bar.

### 📄 Pagination
- Candidate Pipeline, Candidate Profiles, Search & Filters, Job Management,
  and the Admin Panel's candidate list all now page through results
  (25/page) instead of rendering everything at once.

### 🧾 Onboarding / Payroll
- New **"Onboarding / Payroll"** page: once an application's status becomes
  `Joined`, HR creates an employee record here (designation, department,
  salary, joining date, employment type, manager, bank account, tax ID) —
  a simple hand-off point to your real payroll/HRIS system. Employee IDs
  are auto-generated (`EMP-0001`, `EMP-0002`, ...).

### 🐳 Docker & CI/CD
- **`Dockerfile`** + **`docker-compose.yml`**: spins up the app, the
  background worker, and MongoDB together. MongoDB isn't exposed to the
  host by default — only the app/worker containers can reach it.
- **`.github/workflows/ci.yml`**: on every push/PR, runs a syntax check,
  `flake8` (catching real errors, not just style nits), and the full
  `pytest` suite (`tests/`) against an in-memory MongoDB — then builds the
  Docker image to confirm it still builds cleanly.

### 💾 Automated Backups
- **`backup.py`**: uses `mongodump` if available, otherwise falls back to a
  pure-Python JSON export of every collection — either way, old backups
  older than the last N (default 7) are automatically rotated out. Schedule
  it with cron:
  ```bash
  0 2 * * * cd /path/to/ats_portal && /path/to/venv/bin/python backup.py >> backup.log 2>&1
  ```

## 🚀 Setup

### Option A — Local (no Docker)

#### 1. Install MongoDB
- **Local:** install MongoDB Community Server ([mongodb.com/try/download/community](https://www.mongodb.com/try/download/community)) and run `mongod`.
- **Or use MongoDB Atlas** (free tier): create a cluster at [mongodb.com/atlas](https://www.mongodb.com/atlas) and copy its connection string.

#### 2. Configure environment
```bash
cp .env.example .env
```
Edit `.env` and set `MONGO_URI` (local `mongodb://localhost:27017` or your Atlas connection string).

#### 3. Install dependencies
```bash
python -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
pip install -r requirements.txt
```

#### 4. Run the app AND the background worker (two terminals)
```bash
# Terminal 1
streamlit run app.py

# Terminal 2 — required for Bulk Resume Import to actually process
python worker.py
```
Open `http://localhost:8501`. You'll land on the **Career Portal**. Click
**"HR / Staff Login"** to access the backend.

### Option B — Docker (recommended for anything beyond your own laptop)
```bash
cp .env.example .env
# edit .env: set MONGO_ROOT_PASSWORD to something strong
docker compose up -d --build
```
This starts the app (`localhost:8501`), the background worker, and MongoDB
together. Resumes/logos/offers persist in `./assets` on the host; MongoDB
data persists in a Docker volume.

## 📧 Enabling Automatic Emails
Go to **Admin Panel → Branding & AI → SMTP Settings** and save your SMTP
credentials (e.g. Gmail with an App Password — enable 2-Step Verification,
then generate one under Google Account → Security → App Passwords). Once
saved, the system automatically sends:
- **Application Received** — the moment a candidate applies
- **Interview Scheduled** — with a `.ics` calendar invite attached
- **Shortlisted** / **Rejected** — from the **Interview Scheduler → 📬
  Status-based Email Queue**, which classifies candidates by their current
  status so you always know exactly who still needs an email

All templates are editable under **Email Templates** — no code changes needed
to adjust wording.


## 🤖 AI-Powered Resume Screening (Claude / OpenAI / Gemini / Grok)
The Match Score, Missing Skills, Strong Skills, Weak Areas, and Recommendation
can be generated by **any one** of four AI providers — pick whichever you
already have an API key for:

1. Go to **Admin Panel → Branding & AI → AI-Powered Resume Screening**.
2. Pick a provider from the dropdown: **Claude**, **OpenAI**, **Gemini**, or **Grok**.
3. Paste that provider's API key (each is stored separately, so switching
   providers later doesn't make you re-enter the others):
   - Claude → [console.anthropic.com](https://console.anthropic.com)
   - OpenAI → [platform.openai.com](https://platform.openai.com/api-keys)
   - Gemini → [aistudio.google.com/apikey](https://aistudio.google.com/apikey)
   - Grok → [console.x.ai](https://console.x.ai)
4. Click **Save AI Settings** — every new application (single apply or bulk
   import) is now screened by that provider.

If no provider is configured, the API call fails, or it times out (capped at
20 seconds — see note below), the system **automatically falls back** to a
TF-IDF + skill-overlap evaluator. There's always a score; AI is never a hard
dependency for the portal to function.

> **Note on timeouts:** every provider call has a hard 20-second timeout
> (`matching.AI_CALL_TIMEOUT_SECONDS`). This matters because, while building
> this, the older `google-generativeai` package was found to sometimes ignore
> its own per-call timeout on a flaky connection (its gRPC transport retries
> internally regardless) — which would have frozen the Career Portal for
> whoever was applying at that moment. This build uses the newer `google-genai`
> SDK instead, which respects the timeout via a normal HTTP client. All four
> providers are wrapped so a slow/unreachable AI API degrades to the
> rule-based evaluator instead of hanging the UI.

## 🧠 Duplicate Resume Detection
Each parsed resume is hashed (`resume_parser.resume_text_hash`). When a new
resume is submitted (individually or via Bulk Import), `matching.check_duplicate_resume`
checks for an identical hash already on file and flags the candidate record
(`is_duplicate: true`) — useful for spotting the same resume submitted
multiple times (possibly under different emails).

## 🎉 Offer Letters & Joining Confirmation
**Interview Scheduler → 🎉 Offer & Joining Actions** has two tabs:
- **Send Offer Letter**: pick a candidate in `Selected`/`HR Round`, fill in
  designation/salary/joining date, click send — this generates the PDF,
  emails it as an attachment, and **only if the email actually sends
  successfully** automatically moves the application to `Offer Sent` (with
  a confirmation + 🎈 on screen). If the email fails, the status is
  deliberately left unchanged so you don't end up with a candidate marked
  "Offer Sent" who never actually received anything.
- **Confirm Joining**: pick a candidate in `Offer Sent`, send the joining
  confirmation email — on success, status auto-moves to `Joined`. From
  there, go to **Onboarding / Payroll** to create their employee record.

You can also change any candidate's status directly from their
**Candidate Profile** page (useful for corrections or statuses that don't
go through an email step, like `Technical Round` → `HR Round`).

## 📬 About the Status-based Email Queue vs. Offer/Joining Actions
These are two different tools for two different situations:
- **Status-based Email Queue** (Interview Scheduler): for candidates
  **already sitting in** a status (Shortlisted, Rejected, Joined, etc.) —
  usually because you changed their status from the Pipeline table or a
  Candidate Profile — who are still waiting on that stage's email. It just
  sends the matching template; it doesn't change anything.
- **Offer & Joining Actions**: these are the actions that **cause** the
  Selected → Offer Sent → Joined transitions in the first place, because an
  offer letter has to be generated and attached, not just a plain template
  email. That's why they're separate, dedicated flows.

## 🧾 Employee Roster (Onboarding / Payroll)
- **Status** is a fixed dropdown — `Active`, `On Leave`, `Transferred`,
  `Resigned`, `Terminated`, `Retired` — not free text.
- **Department** is a dropdown pulled from the same `departments` collection
  used everywhere else (Job Management, etc.) — nobody can type in a
  department that doesn't already exist.
- Every field **except Employee ID** (`EMP-0001`, `EMP-0002`, ...) is
  editable from **Onboarding / Payroll → Edit Employee Record**.

## 📅 Job Auto-Close on Deadline
Any job whose **"Apply Before"** date has passed is automatically flipped
from `Open` to `Closed` — checked whenever the Career Portal or Job
Management page loads, and again every 60 seconds by `worker.py` as a
safety net (so it happens even on a day nobody visits the portal). This
means expired postings quietly disappear from the public Career Portal
without anyone needing to remember to close them.

## 🔒 Production Hardening Notes
Already handled: bcrypt password hashing, brute-force lockout, forced
password change on the seeded account, stored-XSS escaping, file magic-byte
validation, pagination, TTL-expiring sessions, non-root Docker container,
defense-in-depth RBAC on the Admin Panel, automated backups, and a CI
pipeline. Still worth adding before a large public-internet rollout:
- Put the app behind HTTPS (reverse proxy / load balancer) — Streamlit
  itself doesn't terminate TLS.
- Move SMTP credentials and the Anthropic key to a proper secrets manager
  (e.g. Docker/Kubernetes secrets, AWS Secrets Manager) instead of the
  `settings` collection — fine for a small trusted team, less ideal at
  larger scale with more admins.
- Add a CAPTCHA on the public apply form if it's genuinely internet-facing
  (current file/size/rate protections help, but don't stop a determined bot).
- Consider moving resume file storage to S3/Blob storage instead of local
  disk if you ever run multiple app instances behind a load balancer.
- If you outgrow 5-10 users significantly, swap `worker.py`'s Mongo-backed
  queue for Celery + Redis (more moving parts, but battle-tested at scale).
